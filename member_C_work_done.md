## 1. Features Implemented & Reasoning

### A. The AI Agent Pipeline (`packages/agent/src/modules/`)
*   **Validation Agent (`validation.ts`):** 
    *   *What I did:* Built a deterministic TypeScript engine that cross-checks the seller's, importer's, and logistics partner's inputs (e.g., ensuring weight discrepancies are <5%). 
    *   *Reasoning:* We cannot trust LLMs to do math or enforce hard legal boundaries. I used strict code logic (Symbolic AI) to guarantee 100% legal reliability.
*   **Calculation Agent (`calculation.ts`):**
    *   *What I did:* Built the carbon math engine based on official EU default values (`emissions_factors.json`). 
    *   *Reasoning:* I explicitly separated `cbam_embedded_tco2` (regulatory tax) from `transport_tco2` (internal tracking). This structural separation prevents our users from illegally overpaying carbon taxes on transport emissions.
*   **Analytics Agent (`analytics.ts`):**
    *   *What I did:* Wrote a script that aggregates all historical database rows and sends a prompt to Gemini 1.5 Flash to generate a 2-3 sentence dashboard summary.
*   **Reporting Agent (`report.ts`):**
    *   *What I did:* Wrote the final LLM prompt that generates the markdown report.
    *   *Reasoning:* I injected a **Strict Legal Template** into the LLM prompt. It forces the AI to output a highly formal document containing a "Statement of Purpose", a "Methodology" section proving blockchain immutability, and an official signature block. This stops the AI from sounding like a marketing chatbot.

### B. The Frontend Visualizations (`apps/web/`)

    *   *Reasoning:* Built to target the "SiteLab" hackathon prize by heavily featuring dark-mode and glass-morphism aesthetics.

### C. Testing & Infrastructure

## 2. Cross-Member Merge Coordination (Crucial)

To avoid breaking the app during the merge, Member A and Member B **must** review the following conflict points where my code intersects with theirs.

### ⚠️ Overlaps with Member A (Database & Solana)
My AI Agents read from and write to the Supabase database that Member A designed. 
1.  **Exact Column Names:** My agents expect the columns `cbam_embedded_tco2` and `cbam_exposure_eur` to exist in the `compliance_reports` table. If Member A named these something else (e.g., `total_emissions`), the analytics will crash.
2.  **Foreign Key Relations:** My `analytics.ts` file performs a deep Supabase relational query joining `compliance_reports` ➔ `compliance_bundles` ➔ `seller_attestations`. Member A must ensure these foreign keys are correctly defined in Postgres.
3.  **The Trigger Mechanism:** My agents are currently just pure functions. Member A is responsible for writing the Supabase Edge Function (or Webhook) that actually *calls* my `packages/agent` pipeline the moment a bundle's status changes to `complete` on the blockchain.

### ⚠️ Overlaps with Member B (Next.js Application)
I placed my dashboard and report files into `apps/web/`, which is the directory Member B is actively scaffolding. 
1.  **NPM Dependencies:** I used libraries that are not in the default Next.js installation. Before merging my code, Member B **must** run:
    `pnpm add recharts swr react-markdown` inside `apps/web/`.
2.  **CSS Global Variables:** My components rely on global CSS variables (e.g., `var(--accent-green)`, `var(--bg-glass)`). If Member B chose different names for the glass-morphism tokens in `globals.css`, they must do a find-and-replace on my `.module.css` files to match their theme.
3.  **Routing Integration:** Member B needs to ensure their Sidebar navigation correctly links to `/dashboard` and that their Bundle Detail page has a button linking to `/reports/[id]`, which are the pages I built.
